# Internationalization (i18n)

## Overview

SUNLIGHT's UI is fully localized using **i18next** + **react-i18next**. No user-facing strings are hardcoded in components — all text is stored in translation JSON files.

## Supported Languages

| Code | Language | Direction | Status |
|------|----------|-----------|--------|
| `en` | English | LTR | Complete (canonical) |
| `fr` | French | LTR | Complete |
| `ar` | Arabic | RTL | Complete |
| `es` | Spanish | LTR | Complete |

## Adding a New Language

1. Copy `src/i18n/locales/en.json` to `src/i18n/locales/<code>.json`
2. Translate all values (keys must remain identical)
3. Add the import and entry in `src/i18n/i18n.js`:

```javascript
import xx from './locales/xx.json';

// In SUPPORTED_LANGUAGES array:
{ code: 'xx', name: 'Language', nativeName: 'Native Name', dir: 'ltr' }

// In i18n.init resources:
xx: { translation: xx },
```

That's it. No other code changes needed. The language will appear in the selector automatically.

## Translation Keys

Keys use dot notation: `namespace.key`

```javascript
const { t } = useTranslation();
t('legal.risk_indicator')          // → "Risk indicator, not allegation."
t('risk_inbox.showing', { count: 42, total: 1000 })  // → "Showing 42 of 1000 contracts"
```

### Key namespaces

| Namespace | Content |
|-----------|---------|
| `app` | Application name and tagline |
| `legal` | Legal disclaimers — **handle with care, consult legal before modifying** |
| `nav` | Navigation labels |
| `risk_inbox` | Risk inbox surface |
| `case_packet` | Case packet surface |
| `portfolio` | Portfolio surface |
| `admin` | Administration surface |
| `onboarding` | Onboarding flow |
| `typology` | Fraud typology names |
| `tier` | Risk tier names and descriptions |
| `evidence_standard` | Evidentiary standard names |
| `procurement` | Procurement domain terms |
| `common` | Shared UI elements |
| `formats` | Relative date strings |

### Critical translation: Legal disclaimer

The key `legal.risk_indicator` ("Risk indicator, not allegation.") is a legal requirement. When translating:

- **Do not weaken the meaning.** The translation must convey that this is a statistical indicator, not an accusation.
- **Do not paraphrase loosely.** Keep the structure: "[risk term], [not] [allegation term]."
- **Have institutional legal review** the translation in each language before deployment.

## Locale Formatting

All dates, numbers, and currencies use browser-native **Intl** APIs. No manual formatting.

```javascript
import { formatCurrency, formatDate, formatPercent } from '../utils/localeFormat';

formatCurrency(1500000, 'XOF', 'fr-FR')  // → "1 500 000 XOF"
formatCurrency(1500000, 'USD', 'en-US')  // → "$1,500,000"
formatDate(new Date(), 'fr-FR', 'Africa/Ouagadougou')  // → "22 févr. 2026"
formatPercent(0.72, 'en-US')             // → "72%"
```

### Tenant formatter

The `createTenantFormatter()` function creates a bound formatter from the tenant profile, so you don't pass locale/currency/timezone to every call:

```javascript
const fmt = createTenantFormatter(tenantProfile);
fmt.currency(1500000)       // Uses tenant's currency and locale
fmt.date(new Date())        // Uses tenant's timezone
fmt.percent(0.72)           // Uses tenant's locale
```

## RTL Support

RTL languages (Arabic, Hebrew, Farsi, Urdu) are detected from the language tag. When an RTL language is active:

1. `<html dir="rtl">` is set automatically
2. CSS logical properties should be used (`margin-inline-start` instead of `margin-left`)
3. Layout direction flips via the `dir` attribute

The `isRTL` flag is available from both `useTenantProfile()` and `useLanguage()` hooks.

## Runtime Language Switching

Language changes take effect immediately without page reload:

```javascript
const { changeLanguage } = useLanguage();
changeLanguage('fr');  // UI updates instantly
```

Priority: user override (localStorage) → tenant default → browser language → English

## Typology IDs

Typology names are translatable but have stable internal IDs for logs and audit trails:

| Internal ID | English | French |
|------------|---------|--------|
| `price_anomaly` | Price Anomaly | Anomalie de prix |
| `vendor_concentration` | Vendor Concentration | Concentration de fournisseurs |
| `timing_anomaly` | Timing Anomaly | Anomalie de timing |
| `split_contract` | Split Contract | Fractionnement de contrat |
| `geographic_clustering` | Geographic Clustering | Regroupement géographique |
| `agency_risk` | Agency-Level Risk | Risque au niveau de l'agence |

The internal ID is always used in API responses, database records, and audit logs. The translated name is only for UI display.
